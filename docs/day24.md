**Q1. 다음 중 엔터티(Entity)의 분류에 대한 설명으로 가장 적절하지 않은 것은?**
① 발생 시점에 따라 기본/키 엔터티, 중심 엔터티, 행위 엔터티로 구분할 수 있다.
② 행위(Active) 엔터티는 두 개 이상의 부모 엔터티로부터 발생하며, 데이터량이 가장 많고 수시로 변경되는 특징을 가진다.
③ 사건(Event) 엔터티는 업무 수행에 따라 발생되는 엔터티로, 주문, 청구, 미수금 등이 이에 해당한다.
④ 유형(Tangible) 엔터티는 물리적인 형태가 없고 개념적으로만 존재하는 엔터티로, 조직, 보험상품 등이 해당한다.

* **정답:** ④
* **힌트:** '유형(Tangible)'이라는 단어의 의미(형태가 있음)를 떠올려 보십시오.
* **해설:** 물리적인 형태가 없고 개념적으로만 존재하는 엔터티는 '개념(Conceptual) 엔터티'입니다. 유형 엔터티는 사원, 물품, 강사 등 물리적 형태가 명확한 엔터티입니다.

**Q2. 다음 속성(Attribute)에 대한 설명 중, 데이터 무결성 확보를 위해 설계 시 가급적 생성을 지양해야 하는 속성은 무엇인가?**
① 원래 업무상 존재하는 기본 속성 (Basic Attribute)
② 업무를 모델링하기 위해 새로 부여된 설계 속성 (Designed Attribute)
③ 다른 속성의 영향을 받아 계산되어 산출되는 파생 속성 (Derived Attribute)
④ 유일한 값을 가지는 단일 값 속성 (Single-Valued Attribute)

* **정답:** ③
* **힌트:** 원본 데이터가 변경되었을 때 함께 업데이트되지 않으면 치명적인 데이터 불일치가 발생합니다.
* **해설:** 파생 속성은 계산된 값(예: 총합계, 이자율 등)이므로, 원천 데이터가 변경될 때 정합성을 유지하기 위한 추가적인 프로세스(트리거, 배치 등)가 필요하여 가급적 피하는 것이 좋습니다.

**Q3. 관계(Relationship)를 정의할 때 고려해야 할 3가지 핵심 요소로 바르게 짝지어진 것은?**
① 관계명(명칭), 관계차수(기수성), 관계선택사양(필수/선택)
② 관계명(명칭), 식별자(PK), 관계차수(기수성)
③ 부모엔터티, 자식엔터티, 외래키(FK)
④ 관계차수(기수성), 관계선택사양(필수/선택), 도메인(Domain)

* **정답:** ①
* **힌트:** 관계를 ERD에 그릴 때 선의 이름, 까마귀발(1:N), 동그라미(선택) 기호를 생각하십시오.
* **해설:** 관계를 정의할 때는 관계명(Relationship Name), 관계차수(Degree/Cardinality - 1:1, 1:N, M:N), 관계선택사양(Optionality - 필수, 선택)의 3요소를 반드시 정의해야 합니다.

**Q4. 비식별자 관계(Non-Identifying Relationship)를 설정하는 것이 더 적절한 상황으로 가장 알맞은 것은?**
① 부모 엔터티의 인스턴스가 소멸될 때 자식 엔터티의 인스턴스도 반드시 함께 소멸되어야 하는 경우
② 자식 엔터티의 주식별자를 구성하는 속성 수가 너무 많아져 SQL 질의가 지나치게 복잡해질 우려가 있는 경우
③ 부모 엔터티와 자식 엔터티의 생명주기(Life Cycle)가 완벽하게 일치하는 경우
④ 자식 엔터티가 부모 엔터티의 식별자를 통해서만 존재 의미를 가지는 강한 종속성을 띨 경우

* **정답:** ②
* **힌트:** 식별자 관계로 계속 연결하면 PK 속성이 누적되어 늘어납니다.
* **해설:** 부모의 PK를 자식의 PK로 계속 상속하는 식별자 관계는 계층이 깊어질수록 자식의 주식별자 속성 수가 증가하여 조인(Join) 문장이 복잡해집니다. 이 경우 비식별자 관계를 통해 부모의 식별자를 일반 속성(FK)으로만 가지게 하여 구조를 단순화하는 것이 유리합니다.

**Q5. 다음 중 BCNF(Boyce-Codd Normal Form) 정규화에 대한 설명으로 올바른 것은?**
① 모든 속성이 원자값을 가지도록 다중 값 속성을 분해하는 과정이다.
② 부분 함수 종속성을 제거하는 과정이다.
③ 일반 속성 간의 이행적 함수 종속성을 제거하는 과정이다.

* **정답:** ④

* **해설:** BCNF는 제3정규형(3NF)을 만족하는 릴레이션에서, 일반 속성이 주식별자의 일부를 결정하는 등 '모든 결정자가 후보키'가 되도록 릴레이션을 분해하는 보다 엄격한 정규화 단계입니다.

* **해설:** 전체 데이터를 일괄적으로 조회하거나 처리하는 트랜잭션이 빈번할 경우, 조인을 최소화하기 위해 모든 수퍼/서브 타입의 속성을 하나의 테이블로 統合(Single Type)하는 것이 성능상 가장 유리합니다.

**Q11. 다음 [TBL_A]와 [TBL_B]를 대상으로 아래 쿼리를 실행했을 때, 반환되는 총 행(Row)의 수는?**

> **[TBL_A]** (ID: 1, 2, 3)
> **[TBL_B]** (ID: 1, 2)

```sql
SELECT A.ID, B.ID
FROM TBL_A A LEFT OUTER JOIN TBL_B B
  ON A.ID = B.ID AND B.ID = 1;

```

① 1행
② 2행
③ 3행
④ 6행

* **정답:** ③
* **힌트:** `ON` 절의 조건은 조인 대상을 제한할 뿐, `LEFT OUTER JOIN`의 기준 테이블(TBL_A)의 데이터 보존 특성을 무효화하지 않습니다.
* **해설:** 조인 시 B.ID가 1인 데이터만 조인에 참여합니다. 그러나 `LEFT JOIN`이므로 A 테이블의 데이터(1, 2, 3)는 모두 무조건 보존됩니다.
A.ID=1 (B.ID=1 매칭)
A.ID=2 (매칭 안됨, B.ID는 NULL)
A.ID=3 (매칭 안됨, B.ID는 NULL)
따라서 총 3행이 출력됩니다.

**Q12. 위 Q11번과 유사하지만 조건의 위치가 변경된 아래 쿼리의 최종 반환 행(Row)의 수는?**

```sql
SELECT A.ID, B.ID
FROM TBL_A A LEFT OUTER JOIN TBL_B B
  ON A.ID = B.ID
WHERE B.ID = 1;

```

① 1행
② 2행
③ 3행
④ 에러 발생

* **정답:** ①
* **힌트:** `WHERE` 절은 모든 조인이 끝난 후 생성된 임시 결과 집합에 대해 마지막으로 필터링을 수행합니다.
* **해설:** 먼저 `LEFT JOIN`이 수행되어 3행(1-1, 2-2, 3-NULL)이 임시 생성됩니다. 그 후 `WHERE B.ID = 1`이 평가됩니다. 2와 NULL은 이 조건을 통과하지 못하므로, 최종적으로 A.ID=1 인 데이터 단 1건만 반환됩니다. (LEFT JOIN이 사실상 INNER JOIN처럼 동작하게 되는 함정입니다.)

**Q13. 아래 쿼리의 실행 결과로 반환되는 값을 순서대로 바르게 나열한 것은?**

```sql
SELECT NULLIF('A', 'B'),
       NULLIF('C', 'C'),
       COALESCE(NULL, 'D', 'E')
FROM DUAL;

```

① A, NULL, D
② NULL, C, E
③ B, NULL, D
④ A, C, NULL

* **정답:** ①
* **힌트:** `NULLIF`는 같으면 NULL, 다르면 첫 번째 인자를 반환합니다. `COALESCE`는 NULL이 아닌 최초의 값을 반환합니다.
* **해설:** `NULLIF('A', 'B')`는 다르므로 'A' 반환. `NULLIF('C', 'C')`는 같으므로 `NULL` 반환. `COALESCE(NULL, 'D', 'E')`는 첫 번째 NULL이 아닌 'D'를 반환합니다.

**Q14. 다음 [TB_TEST] 테이블에 대하여 DDL과 DML 연산을 수행한 직후의 테이블 상태에 대한 설명으로 옳은 것은? (Oracle 환경)**

```sql
INSERT INTO TB_TEST VALUES (10);
INSERT INTO TB_TEST VALUES (20);
SAVEPOINT SV1;
UPDATE TB_TEST SET VAL = 30 WHERE VAL = 20;
CREATE TABLE TB_TEMP (ID NUMBER);
ROLLBACK TO SV1;

```

① 테이블에는 10과 20이 저장되어 있다.
② 테이블에는 10과 30이 저장되어 있다.
③ `ROLLBACK TO SV1` 명령 수행 시 에러가 발생한다.
④ 테이블의 모든 데이터가 롤백되어 아무 데이터도 없다.

* **정답:** ③
* **힌트:** DDL 명령문이 실행되면 그 즉시 묵시적 커밋(Auto-Commit)이 발생합니다.
* **해설:** 오라클에서 `CREATE TABLE`과 같은 DDL 문장이 실행되면, 그 이전에 수행된 트랜잭션(10, 20 입력 및 30 수정)이 자동으로 `COMMIT` 처리됩니다. 이와 동시에 설정해 둔 `SAVEPOINT`도 소멸하므로, 이후 `ROLLBACK TO SV1`을 호출하면 "저장점이 존재하지 않는다"는 에러가 발생합니다.

**Q15. 다음 `GROUP BY ROLLUP` 쿼리의 실행 결과로 생성되는 소계/총계 레벨의 개수는?**

```sql
SELECT COL1, COL2, COL3, SUM(VAL)
FROM DATA_TAB
GROUP BY ROLLUP(COL1, (COL2, COL3));

```

① 2개
② 3개
③ 4개
④ 8개

* **정답:** ②
* **힌트:** 괄호로 묶인 `(COL2, COL3)`은 논리적으로 하나의 단일 컬럼처럼 취급됩니다.
* **해설:** `ROLLUP(A, B)`는 (A, B) -> (A) -> () 의 3개 레벨을 가집니다. 여기서 B가 `(COL2, COL3)`이므로,

1. (COL1, COL2, COL3) 상세
2. (COL1) 소계
3. () 전체 총계
이렇게 총 3가지 레벨의 집계가 생성됩니다.

* **정답:** ③
* **힌트:** 필요 없는 데이터는 무거운 GROUP BY 작업을 시작하기 전에 `WHERE` 절에서 미리 쳐내는 것이 정석입니다.

**Q33. 다음 중 테이블에 설정할 수 있는 제약조건(Constraints)에 대한 설명으로 틀린 것은?**

③ `FOREIGN KEY` 참조를 받고 있는 부모 테이블의 데이터(PK)는 자식 테이블에 해당 값을 참조하는 데이터가 남아 있어도 언제든 자유롭게 삭제(DELETE)할 수 있다.
④ `CHECK` 제약조건은 특정 컬럼에 입력될 수 있는 값의 범위나 조건을 도메인 규칙처럼 통제한다.

* **힌트:** 부모가 사라지면 고아 데이터가 발생하여 참조 무결성이 깨집니다.
* **해설:** `FOREIGN KEY`로 연결된 경우, 참조 무결성(Referential Integrity) 원칙에 따라 자식 테이블에서 참조하고 있는 부모 데이터는 함부로 삭제하거나 수정할 수 없습니다 (단, `ON DELETE CASCADE` 옵션이 설정된 경우는 예외).

* **해설:** 서브쿼리는 그룹 함수(`MAX`)에 의해 단일 행을 반환하는 것이 보장됩니다. 하지만 메인 쿼리는 급여가 5000인 사원을 찾는 `WHERE` 조건이 되므로, 최고 급여 동점자가 여러 명이라면 메인 쿼리의 최종 출력은 다중 행이 됩니다. 따라서 최종 결과가 단일 행이 되려면 최고 급여자가 1명이어야 합니다.

**Q45. 인덱스(Index) 튜닝 및 최적화 관점에서 인덱스 스캔을 무력화시키고 테이블 전체 스캔(Full Table Scan)을 유발하는 대표적인 안티 패턴(Anti-Pattern)이 아닌 것은?**
① 인덱스가 생성된 대상 컬럼을 `WHERE SUBSTR(EMPNO, 1, 2) = '10'`과 같이 외부 함수로 가공하는 경우
② 인덱스 컬럼에 대해 `WHERE SAL * 1.1 > 5000`과 같이 산술 연산을 수행하는 경우
③ 문자형으로 선언된 인덱스 컬럼에 숫자형 리터럴 값을 비교하여 암시적 형변환이 발생하는 경우
④ 복합 인덱스(A, B 컬럼)가 구성되어 있을 때 선행 컬럼(A)을 `WHERE A = 'VALUE'` 형태로 정상 비교하는 경우

* **정답:** ④
* **힌트:** 좌변(인덱스 컬럼)을 훼손하면 인덱스라는 사전의 가나다순 정렬 체계가 무너져 색인을 사용할 수 없습니다.

② UNION ALL

**Q48. `DELETE` 명령과 `TRUNCATE` 명령의 차이를 설명한 것으로 올바르지 않은 것은?**
① `DELETE`는 DML이므로 데이터베이스 버퍼를 통해 한 건씩 지우며 Undo 공간을 사용하여 롤백이 가능하다.
② `TRUNCATE`는 DDL이므로 데이터가 통째로 잘려나가듯 즉각 삭제되며 원칙적으로 롤백할 수 없다.
③ `DELETE` 시 `WHERE` 조건을 생략하면 테이블 내의 모든 데이터가 삭제되며 테이블의 스키마 구조까지 함께 드롭(Drop)된다.
④ `TRUNCATE` 수행 시 인덱스 트리 구조와 할당되었던 디스크 세그먼트 공간까지 초기화된다.

* **정답:** ③
* **힌트:** 데이터를 모두 지우는 것과 데이터베이스 객체 껍데기(테이블 구조) 자체를 삭제하는 것은 완전히 다른 차원의 명령입니다.
* **해설:** `DELETE` 명령어에 `WHERE` 절을 생략하면 테이블의 모든 행을 삭제하지만, 테이블의 논리적 구조(컬럼 정의 등 스키마 형태)는 그대로 남아 데이터 삽입 대기 상태를 유지합니다. 구조까지 완전히 삭제하는 것은 `DROP TABLE` 명령어입니다.

**Q49. 오라클의 `DECODE` 함수와 ANSI 표준의 `CASE` 표현식을 비교한 설명으로 가장 부적절한 것은?**
① `DECODE(COL, A, B, C)`는 `CASE WHEN COL = A THEN B ELSE C END`와 동일한 논리 구조이다.
② `CASE` 구문은 오직 동등(`=`) 비교만 가능하지만, `DECODE` 함수는 부등호(`<, >, BETWEEN`) 비교를 자유롭게 수행할 수 있어 유연성이 높다.
③ `CASE` 구문은 분기 로직이 순차적으로 평가되며, 첫 번째 TRUE 조건을 만족하면 이후 평가는 생략된다.
④ `DECODE`는 오라클 전용 함수이므로 타 DBMS 이식성이 떨어지지만, `CASE` 표현식은 ANSI 표준 문법이므로 대부분의 RDBMS에서 호환된다.

* **정답:** ②
* **힌트:** 두 구문의 유연성을 거꾸로 설명하고 있습니다. 함수 인자로 부등호를 넣을 수 있는지 생각해 보십시오.
* **해설:** `DECODE` 함수는 내부적으로 정확한 동등 일치(`=`) 비교만을 수행하도록 설계된 단순 분기 함수입니다. 부등호나 논리 연산자(`AND`, `OR`), 복잡한 조건 수식은 오직 `CASE` 표현식에서만 자유롭게 구현할 수 있습니다.

**Q50. 데이터베이스 뷰(View) 객체에 데이터를 삽입/갱신(DML)할 때 발생하는 제약사항으로 가장 알맞은 것은?**
① 뷰는 가상 테이블이므로 어떠한 상황에서도 뷰를 통한 DML 연산은 절대 불가능하다.
② 원본 테이블의 기본키(PK)나 `NOT NULL` 제약이 걸린 필수 컬럼이 뷰의 `SELECT` 구문에 포함되어 있지 않더라도, 뷰를 통해 `INSERT`를 성공적으로 수행할 수 있다.
③ 단일 테이블 기반의 단순 뷰(Simple View)이고 산술식이나 집계 함수 처리가 포함되지 않은 경우, 원본 테이블의 제약조건을 위배하지 않는다면 뷰를 통한 제한적인 DML 연산이 가능하다.
④ 다수의 테이블이 `JOIN`으로 결합된 복합 뷰(Complex View)는 항상 모든 갱신 연산을 정상적으로 원본 테이블들에 전파(Cascade)한다.

* **정답:** ③
* **힌트:** 뷰는 '투명한 거울'일 때만 거울 너머 원본 실체를 만질 수 있습니다. 여러 겹으로 꼬인 거울(Join)이거나 일그러진 거울(집계/산술 연산)일 경우 조작이 불가합니다.
* **해설:** 뷰를 통해 데이터를 갱신(Update, Insert, Delete)하려면 뷰가 원본 테이블의 특정 행을 완벽히 1:1로 가리킬 수 있어야 합니다. 집계 함수(GROUP BY), DISTINCT, ROWNUM, 산술식 가공, 조인 등이 포함된 뷰는 논리적으로 특정 행을 역추적할 수 없으므로 뷰를 통한 DML 연산이 차단됩니다. 또한 원본의 `NOT NULL` 필수 컬럼이 뷰에 누락되어 있으면 `INSERT`가 불가능합니다.
